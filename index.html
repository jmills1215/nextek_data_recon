Process Overview

Welcome to the comprehensive training guide for Salesforce data reconciliation. This process is designed to streamline data quality management for FMO/IMO clients uploading financial service provider information.

Goal: Create a systematic process where Nextek's dev team handles initial data analysis and CSV preparation, while data admin performs final Data Loader uploads.

Pipeline Workflow

1Salesforce Reporting: Run reports to identify data quality issues (save to "BI Specific" folder)

2Duplicate Detection: Use SF tools and custom SOQL queries to find duplicates

3Error Research: Investigate and categorize data inconsistencies using SOQL verification

4CSV Preparation: Build clean, upload-ready CSV files with SOQL validation

5Handoff: Package files for Data Loader upload with verification reports

FMO/IMO Specifics: These organizations primarily use Person objects in Salesforce, creating unique challenges with contact data, licensing information, and hierarchical relationships.

Salesforce Report Building Fundamentals

Report Types Overview

Tabular Reports

Simple list format - best for data exports and basic analysis

  • Fastest to create and run
  • Good for CSV exports
  • Limited grouping options
  • Ideal for: Data exports, simple lists, record counts

Summary Reports

Grouped data with subtotals - most versatile for analysis

  • Up to 3 levels of grouping
  • Subtotals and grand totals
  • Chart compatibility
  • Ideal for: Duplicate detection, trend analysis, dashboards

Matrix Reports

Cross-tab format with row and column grouping

  • Two-dimensional grouping
  • Complex data relationships
  • Advanced summarization
  • Ideal for: Complex duplicate detection, cross-reference analysis

SOQL-Based Problem Detection

Best Practice: Use SOQL queries in addition to reports for comprehensive problem detection. SOQL provides more precise control and verification capabilities.
# Advanced SOQL for comprehensive problem detection Multi-Field Duplicate Detection: SELECT FirstName, LastName, Email, Phone, COUNT(Id) as DupeCount FROM Contact WHERE Email != null OR Phone != null GROUP BY FirstName, LastName, Email, Phone HAVING COUNT(Id) > 1 ORDER BY DupeCount DESC Data Completeness Analysis: SELECT COUNT(Id) as TotalRecords, COUNT(Email) as HasEmail, COUNT(Phone) as HasPhone, COUNT(LastName) as HasLastName FROM Contact WHERE RecordType.Name = 'Producer' Relationship Integrity Check: SELECT COUNT(CASE WHEN AccountId != null THEN 1 END) as WithAccount, COUNT(CASE WHEN AccountId = null THEN 1 END) as Orphaned FROM Contact WHERE RecordType.Name = 'Agent' Recent Data Quality Issues: SELECT Id, Name, Email, Phone, CreatedDate FROM Contact WHERE CreatedDate >= LAST_N_DAYS:7 AND (Email = null OR Phone = null OR LastName = null)

Duplicate Detection Methods

Salesforce Native Tools

Duplicate Rules

Use Account and Contact duplicate rules to identify potential matches based on:

  • Email address (exact match)
  • First Name + Last Name + Phone
  • SSN or Tax ID (if available)
  • License numbers

Advanced Duplicate Detection Query

SELECT Email, FirstName, LastName, COUNT(Id) as DupeCount FROM Contact WHERE Email != null GROUP BY Email, FirstName, LastName HAVING COUNT(Id) > 1 ORDER BY DupeCount DESC

SOQL Verification for Duplicate Detection

Critical Step: Always verify duplicate detection results with SOQL queries before creating CSVs. This ensures accuracy and prevents false positives.
# SOQL queries to verify duplicate detection Email Duplicates Verification: SELECT Email, COUNT(Id) as RecordCount, MAX(Name) as SampleName FROM Contact WHERE Email != null GROUP BY Email HAVING COUNT(Id) > 1 ORDER BY RecordCount DESC Phone Duplicates Check: SELECT Phone, COUNT(Id) as RecordCount, MAX(Name) as SampleName FROM Contact WHERE Phone != null AND Phone != '' GROUP BY Phone HAVING COUNT(Id) > 1 Cross-Reference Person Accounts: SELECT PersonEmail, COUNT(Id) as RecordCount FROM Account WHERE IsPersonAccount = TRUE AND PersonEmail != null GROUP BY PersonEmail HAVING COUNT(Id) > 1 Verify Record IDs for Updates: SELECT Id, Email, Name FROM Contact WHERE Email IN ('duplicate1@email.com', 'duplicate2@email.com')
Remember: Save all duplicate detection reports to the "BI Specific" folder in Salesforce when creating them through the Reports interface.

FMO/IMO Specific Duplicate Scenarios

Common Issues:
  • Same agent appears under multiple agencies
  • Personal vs. business contact information conflicts
  • Multiple license records for same individual
  • Spelling variations in names (Bob vs. Robert)
  • Multiple email addresses (personal/business)

Duplicate Resolution Process

Step-by-Step Resolution

  • Identify master record (most complete data)
  • Document differences between duplicates
  • Check for unique identifiers (SSN, License #)
  • Review relationship history and activities
  • Prepare merge or deletion CSV
  • Flag for manual review if uncertain

Error Research and Investigation

Common Data Validation Errors

Format Validation Errors

  • Email: Invalid format, missing @ symbol
  • Phone: Wrong length, invalid characters
  • ZIP Code: Invalid format for country
  • Date Fields: Future dates, invalid formats

Required Field Violations

  • Missing Last Name on Person Accounts
  • Empty Record Type assignments
  • Missing Account ownership
  • Incomplete license information

Investigation Workflow

Error Analysis Steps: 1. Export error report from Salesforce (save to "BI Specific" folder) 2. Categorize errors by type and severity 3. Use SOQL queries to verify error patterns 4. Research source data for corrections 5. Document patterns and root causes 6. Prepare correction CSV with proper formatting 7. Verify corrections with additional SOQL queries 8. Test small batch before full upload

SOQL Verification for Error Research

Data Verification Essential: Use SOQL queries to validate error patterns and confirm corrections before creating CSV files. This prevents propagating data issues.
# SOQL queries for error verification Missing Required Fields Check: SELECT Id, Name, Email, Phone, LastName FROM Contact WHERE (Email = null OR Email = '') OR (LastName = null OR LastName = '') OR (Phone = null OR Phone = '') Invalid Email Format Detection: SELECT Id, Name, Email FROM Contact WHERE Email != null AND (NOT Email LIKE '%@%' OR Email LIKE '%@@%' OR Email LIKE '% %') Phone Format Issues: SELECT Id, Name, Phone FROM Contact WHERE Phone != null AND (LEN(Phone) < 10 OR LEN(Phone) > 15 OR Phone LIKE '%555-0199%') Orphaned Records Verification: SELECT Id, Name, AccountId FROM Contact WHERE AccountId = null OR AccountId = '' Verify Fix Before Upload: SELECT Id, Email FROM Contact WHERE Id IN ('003XXXXXXX', '003YYYYYYY') # IDs to be updated

CSV Preparation and Formatting

Data Loader CSV Requirements

CSV Formatting Checklist

  • UTF-8 encoding to handle special characters
  • Column headers match Salesforce API names
  • Record IDs included for updates
  • Date fields in YYYY-MM-DD format
  • Boolean fields as TRUE/FALSE
  • No line breaks within cells
  • Escape commas in text fields

Python CSV Processing Script

import pandas as pd import re from datetime import datetime def clean_salesforce_data(input_file, output_file): # Read the exported data df = pd.read_csv(input_file) # Clean email addresses df['Email'] = df['Email'].str.lower().str.strip() # Format phone numbers (US format) def format_phone(phone): if pd.isna(phone): return '' clean = re.sub(r'[^\d]', '', str(phone)) if len(clean) == 10: return f'({clean[:3]}) {clean[3:6]}-{clean[6:]}' return phone df['Phone'] = df['Phone'].apply(format_phone) # Format names (Title Case) df['FirstName'] = df['FirstName'].str.title() df['LastName'] = df['LastName'].str.title() # Remove duplicates based on email df_clean = df.drop_duplicates(subset=['Email'], keep='first') # Export for Data Loader df_clean.to_csv(output_file, index=False, encoding='utf-8') print(f"Processed {len(df)} records, output {len(df_clean)} clean records") # Usage example clean_salesforce_data('raw_data.csv', 'upload_ready.csv')

Common FMO/IMO Data Issues

Person Object Challenges

Unique Identifier Conflicts

Issue: Multiple agents with same SSN or license number

Resolution: Research with source system, may require manual verification

Prevention: Implement external ID fields for better tracking

Hierarchical Relationship Errors

Issue: Agent assigned to non-existent agency account

Resolution: Create parent accounts first, then assign agents

Prevention: Use external IDs for parent account lookup

License Data Inconsistencies

Issue: Expired licenses, wrong states, missing appointment data

Resolution: Cross-reference with state licensing databases

Prevention: Implement validation rules and regular audits

Financial Services Specific Problems

Compliance Considerations:
  • FINRA registration validation
  • State insurance licensing requirements
  • Privacy regulations for personal data
  • Appointment and contracting status
